## Anti-Patterns Verdict

**LLM assessment:** Not obviously AI-generated, but two patterns undercut the design.

The **callout monoculture** is the primary tell: every content type (objective, hint, guidance, real-world, insight) gets its own tinted-background card with an uppercase label, stacked in a 320px column. The result is a vertical list of identically-proportioned boxes that feel assembled rather than designed. This is the defining hallmark of AI content generation applied to EdTech.

The **vestigial step rail** — a 60px column of 5 non-interactive dots — is the skeleton of a navigation feature that was never implemented. It duplicates the "Step X of 5" eyebrow, provides no click functionality, and consumes permanent horizontal real estate.

The warm-beige palette, Atkinson Hyperlegible/IBM Plex Mono pairing, and color-semantic encoding (teal for Fx, amber for Fy, red for force) are genuine personality choices that elevate the design above the generic EdTech blue default.

**Deterministic scan:** 1 warning from the CLI detector — `single-font` on `index.html`. This is a **false positive**: `style.css` defines both `--font-body: "Atkinson Hyperlegible"` and `--font-mono: "IBM Plex Mono"`, both actively used. The detector scanned the HTML and counted only one Google Fonts preconnect; it missed the CSS token. No genuine anti-patterns remain from the previous run — side-stripe borders and tracked-uppercase eyebrows were correctly resolved in the last polish pass.

**One additional deterministic finding not in the CLI output:** `.num-input { font-size: 0.875rem }` (style.css) resolves to 14px. iOS Safari auto-zooms any `<input>` with `font-size < 16px` on focus — this breaks the layout on iPhone.

## Overall Impression

The simulation has a coherent visual identity and strong interaction mechanics (numeric + slider coupling, live equations, canvas drag). The engineering content is well-written. The fundamental problem is a density mismatch: the guided stepper architecture promises one concept at a time, but the step panel delivers all supporting content simultaneously. A student opening Step 1 faces 5 text callouts before touching a control. Fix that, and the score improves substantially.

---

## What's Working

**1. The numeric input + slider coupling is well-implemented.** Select-on-focus, blur clamping, NaN guard, Enter commits, Shift+Arrow fine-step — all present. The unit label beside the input eliminates unit ambiguity. This is stronger than most educational simulation tools.

**2. The token system produces genuine design coherence.** The color semantic encoding (teal = Fx, amber = Fy, red = applied force) is consistent from CSS variables through canvas geometry through equation labels. The warm-beige palette reads as "engineering sketchpad," not generic EdTech blue.

**3. The insight callout on Step 5 is the best educational moment in the product.** Live narration keyed to angle thresholds (0°, 45°, 90°, 180°, etc.) connects the visual to the concept in real-time. This is the model the entire tool should follow — live description updating as the student interacts.

## Priority Issues

**[P0] Callout overload collapses the step panel into a wall of text**
- **What**: Steps 1–3 render up to 5 simultaneous content blocks inside a 320px panel: Learning Objective, hint, "What You Are Seeing," guidance, and real-world example — all at similar visual weight and 0.875rem font size. The student has no signal about reading order or which block is most important.
- **Why it matters**: The guided stepper architecture is violated from the first screen. A first-year student opens Step 1 and faces ~200 words before touching any control. Cognitive load assessment: 5 of 8 checklist items fail (single focus, visual hierarchy, one-thing-at-a-time, working memory, progressive disclosure). This is the number-one reason the UI feels encyclopedic rather than guided.
- **Fix**: Step 1 should show: title, lead, ONE callout (hint only), and controls. Move "What You Are Seeing" to a collapsible disclosure or an on-hover tooltip on the canvas. Move real-world to a secondary position revealed only after the student has interacted. Collapse the Learning Objective after Step 1 completes.
- **Suggested command**: `$impeccable distill`

**[P1] Step rail dots are non-interactive and duplicate the step eyebrow**
- **What**: The `.step-rail` is a 60px column of 5 numbered dots with no click handler. They repeat the "Step X of 5" information already visible in the step-eyebrow. An advanced student who wants to jump to Step 4 must click Next four times.
- **Why it matters**: Users expect progress indicators in interactive tools to be navigation controls. Non-interactive dots in a fixed column feel like a placeholder. The rail also fails mobile — it's ordered to the top via `order: -1` on narrow screens, occupying space that could go to the canvas.
- **Fix**: Add click handlers on `.rail-dot` elements to jump directly to that step (call `stepIndex = target; refresh()`). Or remove the rail and replace the step-eyebrow with a horizontal progress bar (5 segments, 4px tall) that collapses gracefully on mobile.
- **Suggested command**: `$impeccable harden`

**[P1] Magnitude F = 0 N renders a blank canvas with no explanation**
- **What**: When the magnitude slider is at 0 N, all vector geometry (arrow, labels, components) disappears. The canvas shows only grid and axes on the background — indistinguishable from a loading state or a rendering error.
- **Why it matters**: A first-year student who accidentally drags the slider to 0 will think the simulation has crashed. The simulation has no "zero force" state — it simply shows nothing. This is especially confusing on Step 2 and 3 where the component arrows should still indicate direction.
- **Fix**: Either clamp magnitude minimum to 1 N (`min="1"` in the slider config), or render a special "F = 0 N — no force applied" annotation at the origin. Clamping to 1 is simplest and educationally correct (force magnitude of zero is undefined in this context).
- **Suggested command**: `$impeccable harden`

**[P2] Concept-check has no retry path after a wrong answer**
- **What**: On Steps 2 and 3, all option buttons are immediately disabled after the first click. The feedback text appears but there is no "Try again" action. The only recovery is Back → Next to re-navigate to the step, which resets the card state.
- **Why it matters**: For exam preparation, the concept-check's educational value is in the attempt and the explanation. Permanent lock-out after one wrong click feels punitive and removes the repetition needed for recall.
- **Fix**: Add a "Try again" button in the `.cc-feedback` area that re-enables the option buttons and clears the feedback state, allowing one more attempt before the answer is revealed.
- **Suggested command**: `$impeccable harden`

**[P2] iOS Safari auto-zooms on numeric input focus (font-size 14px)**
- **What**: `.num-input { font-size: 0.875rem }` resolves to 14px. iOS Safari auto-zooms any `<input>` with `font-size < 16px` on focus, breaking the layout and requiring a pinch-to-reset.
- **Why it matters**: Engineering students on phones and tablets will encounter this on every slider interaction. The layout jumps on every focus — this is a jarring interruption to the simulation flow.
- **Fix**: Set `.num-input { font-size: 1rem }` in `style.css`, or add `@media (max-width: 768px) { .num-input { font-size: 1rem; } }`. The visual size change (16px vs 14px) is imperceptible at the input's size.
- **Suggested command**: `$impeccable harden`

**[P2] Shift+Arrow keyboard shortcut is not persistent**
- **What**: The Step 1 hint is the only place the Shift+Arrow 0.5° fine-step shortcut is mentioned. The hint is replaced with new content on Step 2. No tooltip, label, or persistent reference survives to Steps 2–5.
- **Why it matters**: Keyboard-dependent users who read Step 1 and rely on the shortcut lose the reference. The shortcut is also non-obvious — most users expect Arrow alone to move sliders (it does, in 1° steps). The power-user path is only discoverable if the user reads Step 1's hint carefully and retains it.
- **Fix**: Add a persistent one-line keyboard hint in the controls area footer: "⇧ + ← → for 0.5° steps". Or add a `title="Shift + arrow key for 0.5° steps"` attribute on the slider element itself.
- **Suggested command**: `$impeccable polish`

## Persona Red Flags

**Jordan (First-Timer)**
Opens Step 1. The panel presents: Learning Objective block, step title + lead, "What You Are Seeing" callout, hint callout, guidance callout, real-world callout, and two slider controls — all visible before any interaction. Jordan reads ~200 words before touching the magnitude slider. The hint says "Drag the angle slider from 0° to 90°" while the guidance says "Try dragging the arrowhead directly in the diagram" — two competing calls to action on the same screen. Jordan presses Next → to Step 2. A concept-check appears at the bottom. Jordan answers incorrectly. The buttons lock permanently — Jordan must navigate away and back to try again, but nothing in the UI explains this. **Specific failures:** 5-callout stack on Step 1 (`concept-panel`, `hint-callout`, `guidance-callout`, `realworld-callout` simultaneously visible), competing calls to action, no retry affordance on concept-check.

**Sam (Accessibility-Dependent)**
Tabs into the page. Focus reaches the Back button (disabled, opacity 0.4 — may fail WCAG 1.4.3 contrast), then Next. The step counter has `tabindex="-1"` but `.focus()` is never called on step navigation, so focus is not moved to the new content after pressing Next. The canvas is `aria-hidden="true"` — the entire 3D vector diagram is invisible to screen readers. The equations area has `aria-live="polite"` and is the closest substitute, but it provides numeric values only, not a geometric description. **Specific failures:** No focus management on step change; `aria-hidden` canvas with no equivalent dynamic description; disabled Back button at reduced opacity may not meet contrast requirements.

**Alex (Power User)**
Wants Step 4 immediately. Rail dots are not clickable — must press Next four times. On Step 5, tries entering 22.5° in the angle input. The value commits and the canvas updates correctly, but on blur, `Math.round(v)` in the blur handler rounds 22.5 to 23, discarding the sub-integer precision. The input now shows 23° despite the student entering 22.5°. In an engineering context, precision matters — 22.5° is a meaningful angle (half of 45°). **Specific failures:** Non-interactive rail dots, blur-rounding discards decimal input precision.

**The Engineering Student (Project-Specific)**
First Statics course. Sets θ = 30° with F = 100 N to check their textbook. Expects Fx = F cos 30° = 86.6 N. The equation display shows `Fx = F cos θ = 87 N` (rounded to integer). The student compares with their calculator (86.6 N) and wonders if the simulation is wrong. No disclosure explains that displayed values are rounded to the nearest Newton. **Specific failure:** Integer rounding throughout the display with no "±0.5 N" or "rounded" disclosure creates confusion for students verifying textbook calculations.

## Minor Observations

- `.hint-callout::before { content: "ℹ" }` renders as emoji on macOS and as a text character on Windows — inconsistent cross-platform appearance. Replace with an inline SVG icon.
- On Step 1, when Back is `display:none`, the Next button does not grow to fill the full width on narrow screens. `.btn-primary` has no `flex: 1` or `width: 100%` fallback for the single-button state.
- The `.step-in` animation (`translateY(6px) → 0`) implies downward motion = forward. When pressing Back, the animation still runs in the "forward" direction — a subtle mismatch between motion and semantic meaning.
- `setSliderValues`, `updateReadouts`, and `buildStepCard` all exist, but there is no single `renderState()` function that chains them. If a third caller is added (e.g., URL hash restoration), each sync must be manually chained. Minor architecture note.
- The `equations-area:empty { display: none }` CSS rule is correct and avoids phantom flex gaps on Steps 1–3. This pattern is not extended to `guidance-callout` — if guidance is null, the JS does `style.display='none'` in buildStepCard, which requires a JS round-trip where CSS could handle it.

## Questions to Consider

1. **What if the insight callout were the primary narrator on every step, not just Step 5?** The `_updateInsight` pattern — live text updating as the student drags — is the most effective teaching mechanism in the tool. Steps 1–4 currently use static callouts to explain the same concepts. What would it cost to move "What You Are Seeing" from a static box to a live, drag-responsive description?

2. **Should "Back" have an animation that mirrors "Next" in reverse?** A `translateY(-6px) → 0` transition for Back would reinforce the directional metaphor (Next = move forward/down, Back = move backward/up). Motion with semantic meaning reduces cognitive load; motion without semantic meaning adds it.

3. **What is the rounding policy, and is it documented?** Values are displayed as integers (Math.round to nearest N, nearest degree). For exam-prep use, 86.6 N vs 87 N is a meaningful difference. Is the simulation intended as a conceptual visualizer (integers acceptable) or a calculation verifier (needs 1 decimal)? The answer changes the equation display format, the insight callout's threshold language, and the input step precision.
